Design Principle Methods Team 15 Final Project

The goal of this project was to build robot using the lego Mindstorms NXT parts and make it accomplish a set of tasks, requirements. The flow of the competition is started off by informing our robot of the layout of the arena. Using the Wifi class, which is provided, the robot will be sent the map. To allow the user to input various values. Next, the robot must localize itself in the arena. For this, it needs to find two sets of variables, orientation and position. The orientation is found with the help of the FallingEdgeUSLocalization class. Next, the position are filled in using the LightLocalization class which fills in the X and Y of the robot in the arena. To allow the robot to keep track of its position all along the competition our first thread is implemented, an odometer with the Odometer class. A thread is dedicated to the odometer as one does not want the odometer to lose track of the position and orientation due to another method running. To then allow the user to get feedback on the status of the robot, another thread was used in OdometryDisplay class as we want the screen to be constantly updated whilst the robot performs other actions. Next, our third thread is dedicated to the class OdometryCorrection. This class takes care of correcting the odometer of the robot as the competition unfolds. Odometry correction occurs through the detection of grid lines drawn on the arena. This class must absolutely be a thread as we would not want to miss a line capture whilst the robot moves along the tiles. These three class make up the threads implemented in our software.

Apart from the previously mentioned threads, the class that controls the sequences of actions of the robot is the Controller class. This class implements the logic of the movement of the robot throughout the map by controlling the remaining classes such as the LightLocalization class, the FallingEdgeUSLocalization class, the ZiplineTraversal class, the RiverTraversal class, and the BlockDetection class. The reason behind implementing a separate class that handles all the control of the robot's movements instead of doing it in CaptureMainFlag itself was because we wanted the CaptureFlagMain thread to simply instantiate variables and start the other threads. Besides this, the Navigation class is one that takes care of moving the robot across various environments, whether it be a gridded floor, a zipline, or an imaginary river in the middle of the map. When the terrain happens to be a zipline, or a river, the classes ZiplineTraversal, and RiverTraversal are respectively called aid the navigation. These classes therefore help the robot cross from one side of the map to the other. The class, BlockDetection, then helps with finding blocks and identifying which of them is the opponents flag. Once the flag has been captured, the robot must return to its side with the help of the Navigation class once again. All along, the Controller makes sure that the robot is not running into anything with the help of ObstacleAvoidance class in case the robot gets too close to any object.

The last part of the structure of this project are the interfaces: LightController and UltrasonicController. These interfaces are the basic methods that are needed to get the data from the sensors. Therefore, various classes implement them and possibly override their methods to tailor the interpretation of the data according to their needs.

Below explains the usage of the shared variable amongst the threads followed by a diagram showing their communication.

Odometer's position:
OdometryDisplay Thread needs to talk to the Odometer Thread to get its position to display it.
OdometryCorrection Thread sets the odometer's position during correction (when a line is encountered).

lineEncountered & isLocalizing:
OdometryCorrection Thread is constantly checking if lineEcountered is true AND if isLocalizing is false. Only then will it set the odometer's position.
